## Vision

### 10x Check
NestTrue is not a better research website — it's the product that makes the broker irrelevant. Rahul is in the parking lot of a site visit. He messages @NestTrue on WhatsApp. "Just saw Ramky One Odyssey, 3 BHK, budget ₹1.3Cr, need possession by Jan 2028, WFH." NestTrue responds in 90 seconds: "Caution. Ramky slipped delivery on 3 of last 5 projects. At current construction pace, Jan 2028 is tight. Three alternatives with better delivery track records at your budget: Aparna Serene Park, MyHome Mangala, Auro Regent. Want the comparison?" Rahul feels like he has a knowledgeable friend with no financial stake in his decision. That's the Platonic ideal: The Wirecutter of Indian real estate.

### Platonic Ideal
Every buyer in India's top 5 cities gets a specific, evidence-backed verdict via WhatsApp within minutes of a site visit. Verdicts are verifiable (RERA-linked), accountable (track record of past predictions), and structurally independent (revenue doesn't influence which projects NestTrue recommends). Resident intelligence from verified locals supplements the data layer. NestTrue becomes the default trusted advisor for the ₹80L–3Cr first-time buyer — and eventually, the NRI diaspora.

## Spec Review Fixes (Round 2)

### Admin dashboard (N2)
Admin dashboard EXISTS at `/admin/leads` — password-gated with cookie auth. It displays all leads with phone, city, area, WhatsApp status, score, and qualified flag. Phase 2 adds: failed WhatsApp send indicator (`waStatus: 'failed'`), monthly RERA audit checklist (static checklist page at `/admin/rera-audit`), and a simple "converted" toggle per lead for weekly founder review.

### Inngest delta — Phase 2 additions (issue 11)
Currently fires:
- `lead/created` (from `/api/leads` POST) — triggers WhatsApp intro message to new lead
- `lead/message-received` (from WATI webhook) — appends message to raw transcript, triggers scoring

Phase 2 adds a new step to `lead/message-received`:
1. Accumulate messages until buyer states a project name or area + budget
2. Call Anthropic SDK with: `[area data for stated area]` + `[project data for stated project]` + `[buyer budget, timeline, purpose from lead record]`
3. Generate verdict message (Buy/Caution/Avoid + reason + alternatives)
4. Send verdict back via WATI API
5. Update lead record: `score`, `scoreReason`, `qualified`, `waStatus: 'verdict-sent'`

### Revenue model (N3)
Revenue model to be defined by founder before "How we make money" page is written. Pending decision: flat referral fee vs. % of transaction vs. subscription. Plan reserves the URL `/about/how-we-earn` — content to be authored before Phase 1 ships. This is a founder task, not an engineering task.

### OG image spec (N1)

## Outside Voice Findings & Resolutions

### Finding 1 — Verdict field clarification
Add `verdictBadge: "Buy" | "Caution" | "Avoid"` as a SEPARATE field from existing `verdict: string` (prose) in `types/project.ts`. The prose `verdict` field stays for the full-paragraph analysis. The badge uses `verdictBadge`. Phase 1 task: update type + hand-curate values for all 14 projects + update project page component.

### Finding 2 — RERA verification is manual work, not a 30-min task
9 of 14 RERA numbers are placeholders. Lookup at rera.telangana.gov.in is manual per project. Estimate: 4-6 hours total. Risk: any placeholder that ships to production and gets verified by a buyer's lawyer is a trust-destroying failure. Decision: do ALL 14 before Phase 1 ships. Non-negotiable.

### Finding 3 — LeadCTA component copy fix
The LeadCTA "No spam. No broker referrals. Just honest answers." footer line must also be updated in Phase 1. Updated text: "No spam. We earn a referral fee only when we connect you with a developer — never from brokers." This is a component-level fix in `components/LeadCTA.tsx`.

### Finding 4 — WhatsApp AI data access architecture
Phase 2: inject all relevant area + project data into the Anthropic prompt at call time. Only inject the stated area's data (not all 14 projects). Estimated context: ~800-1,200 tokens per area + project pair. If buyer states a project not in the database: use the area data for that micro-location. Cap prompt at 3,000 tokens. Response cap: 500 tokens (verdict + reason + 3 alternatives).

### Finding 5 — Inngest infrastructure (CRITICAL)
Accepted to Phase 1 scope. Phase 1 adds:
- `lib/inngest.ts` — Inngest client initialization
- `app/api/inngest/route.ts` — serve handler for Inngest functions
- `inngest/functions/lead-qualify.ts` — existing `lead/created` function (move from fire-and-forget)
- `inngest/functions/lead-message.ts` — existing `lead/message-received` handler (move from fire-and-forget)
Phase 2 then adds: AI verdict step to `lead-message.ts`.

### Finding 6 — Content staleness is a current legal risk
Add a visible "Last verified: [date]" timestamp to every project page (already in data as `lastUpdated`). Add disclaimer below possession date: "Possession dates are estimates based on RERA filing — verify current status with developer before booking." This is a 30-minute copy + component change.

### Finding 7 — NRI guide FEMA disclaimer
NRI guide page must include: "This guide covers real estate research only. NRI property purchase in India involves FEMA compliance, RBI restrictions, and tax implications. Consult a CA and FEMA-qualified lawyer before transacting." Standard disclaimer, not legal advice. Added to NRI page spec.

### Finding 8 — Conversion metrics
Add basic event tracking before Phase 1 ships. Use `console.log` + server-side logging initially (already have access to the admin panel). Track: CTA form views, phone number submissions, WhatsApp first message received. Instrument in `app/api/leads/route.ts` and the WATI webhook. No third-party analytics tool required.
